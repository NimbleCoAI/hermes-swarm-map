# Plan: Opinionated Default Config for New Hermes Deployments

**Status:** Research / RFC
**Date:** 2026-05-20
**Author:** Juniper (via research agent)

## Problem

When Swarm Map creates a new Hermes agent (via `POST /api/setup/deploy` or
`POST /api/harnesses/create`), the scaffolded config is minimal — a bare
`config.yaml` with just `model.provider` and `model.default`, a near-empty
`.env`, and a placeholder `SOUL.md`. Users immediately hit known issues:
logging invisible in `docker logs`, no STT configured, no compression, no
session-reset policy, no security hardening. Every new agent requires the
same manual tuning.

The fix: ship an **opinionated default `config.yaml`** that solves known
problems out of the box, and generate a richer `.env` with commented
documentation for platform-specific settings.

## Current State

### How agents are created today

Two paths exist, both in `hermes-swarm-map`:

1. **`/api/harnesses/create` (overlay-only)** — `lib/services/harness.ts`
   `scaffoldAgentDir()` writes a 6-line `config.yaml`:
   ```yaml
   model:
     provider: anthropic
     default: claude-sonnet-4-5
   ```
   And a `.env` with just `ANTHROPIC_API_KEY=` and `API_SERVER_PORT=`.

2. **`/api/setup/deploy` (full deploy)** — `app/api/setup/deploy/route.ts`
   `generateConfigYaml()` writes the same minimal config. `generateEnvContent()`
   is richer (includes platform tokens, Signal URL) but still has no
   config.yaml coverage for logging, STT, compression, etc.

### Files a Hermes agent needs (`~/.hermes-{name}/`)

| File | Purpose | Created today? |
|------|---------|----------------|
| `.env` | Secrets + env overrides | Yes (minimal) |
| `config.yaml` | All runtime config | Yes (3 lines) |
| `SOUL.md` | Persona / system prompt | Yes (placeholder) |
| `memories/` | Persistent memory store | Yes (empty dir) |
| `skills/` | Agent skills | No (created on first use) |
| `logs/` | Log files | No (created at runtime) |
| `sessions/` | Session data | No (created at runtime) |

### Volume mount inconsistency (bug)

`harness.ts` generates compose with `volumes: ${agentDataDir}:/root/.hermes`
but `deploy/route.ts` generates `volumes: ${agentDataDir}:/opt/data`. The
Dockerfile sets `ENV HERMES_HOME=/opt/data` and the entrypoint expects
`/opt/data`. The `/root/.hermes` mount in `harness.ts` is wrong — the
agent's config will not be found because `get_hermes_home()` reads
`HERMES_HOME` (which is `/opt/data` from the Dockerfile).

**This should be fixed as part of this work.** Both paths should mount to
`/opt/data`.

### Missing `command: gateway` in harness.ts

The compose generated by `harness.ts` → `generateStandaloneCompose()` has no
`command:` directive. The Dockerfile's `ENTRYPOINT` runs the entrypoint
script which calls `exec hermes "$@"` — with no args, this drops into the
CLI REPL, not the gateway. The `deploy/route.ts` correctly sets
`command: gateway`. This needs to be fixed in `generateStandaloneCompose()`.

## Known Issues That Sane Defaults Would Solve

### 1. Logging not visible in `docker logs`

**Root cause:** `setup_logging()` in `hermes_logging.py` only adds a stdout
`StreamHandler` when `mode="gateway"`. The gateway does call
`setup_logging(mode="gateway")` so this works — but only if the Python
`logging` module is used. Raw `print()` calls work because
`PYTHONUNBUFFERED=1` is set in the Dockerfile.

**What's missing in config:** There is no config.yaml key that controls
stdout logging — it's hardcoded to activate in gateway mode. This is
actually correct behavior. The real issue users hit is when they try to
debug using `docker logs` and see nothing because the agent exited early
(often due to missing config). The fix here is better error messages in the
entrypoint, not a config default.

**Default to add:**
```yaml
agent:
  verbose: false  # Set true for debug; gateway mode already logs to stdout
```

### 2. STT/TTS not configured

**Root cause:** `stt.enabled` defaults to `true` in the example config, but
the default provider is `"local"` which requires `faster-whisper` to be
installed. The Docker image installs it (via `pip install -e ".[all]"`), so
local STT works in Docker. However, the scaffolded config.yaml has no `stt:`
section at all, so it falls back to code defaults which may or may not
detect the local provider.

**Default to add:**
```yaml
stt:
  enabled: true
  # provider: "local"  # auto-detected; local (faster-whisper) is pre-installed in Docker
  local:
    model: "base"  # tiny|base|small|medium|large-v3|turbo — "base" balances speed/quality
```

### 3. No compression config

Without explicit compression settings, long conversations on messaging
platforms will hit context limits and fail. The example config has sensible
defaults but the scaffolded config has none of them.

**Default to add:**
```yaml
compression:
  enabled: true
  threshold: 0.50
  target_ratio: 0.20
  protect_last_n: 20
  summary_model: "google/gemini-3-flash-preview"
```

### 4. No session reset policy

Messaging sessions grow unbounded without reset config, increasing API costs
with every message. The example config has `session_reset` but scaffolded
agents get nothing.

**Default to add:**
```yaml
session_reset:
  mode: both
  idle_minutes: 1440  # 24 hours
  at_hour: 4          # 4 AM local
```

### 5. No memory config

Persistent memory is a key Hermes feature but isn't activated without config.

**Default to add:**
```yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200
  user_char_limit: 1375
  nudge_interval: 10
  flush_min_turns: 6
```

### 6. No agent behavior limits

Without `max_turns`, agents can run indefinitely on expensive models.

**Default to add:**
```yaml
agent:
  max_turns: 60
  reasoning_effort: "medium"
```

### 7. Signal platform gotchas

Signal requires `SIGNAL_HTTP_URL` (pointing to signal-cli-rest-api) and
`SIGNAL_ACCOUNT`. Inside Docker, the URL must use
`http://host.docker.internal:8080` not `localhost`. The deploy route handles
this correctly but the overlay/create path doesn't set Signal vars at all.

Additionally, `SIGNAL_ALLOWED_USERS` and `SIGNAL_GROUP_ALLOWED_USERS`
default to empty (deny all). New agents should document this clearly.

### 8. Telegram platform gotchas

`TELEGRAM_ALLOWED_USERS` defaults to empty which means **deny all** — new
agents appear dead because they ignore every message. This is a security
default but confusing for setup. Should be documented with a comment.

### 9. No platform toolsets

The scaffolded config doesn't include `platform_toolsets`, so agents fall
back to code defaults. These are generally fine, but making them explicit
prevents surprises.

### 10. Missing `group_sessions_per_user`

Defaults to `true` in the example but absent from scaffolded config. This
is a security-relevant setting — should be explicit.

## Proposed Implementation

### Phase 1: Fix bugs in compose generation

1. **Fix volume mount in `harness.ts`**: Change `/root/.hermes` to `/opt/data`
   in `generateStandaloneCompose()`.
2. **Add `command: gateway`** to `generateStandaloneCompose()`.

These are bugs, not opinions — they should be fixed regardless of whether
the opinionated config ships.

### Phase 2: Create a config template

Add a new file `lib/templates/config.yaml.ts` that exports a function:

```typescript
export function generateDefaultConfig(params: {
  provider: string
  primaryModel: string
  fallbackModel?: string
}): string
```

This generates a full `config.yaml` with all the opinionated defaults
listed above, plus the caller's model choices. Comments explain each
section so users can customize later.

### Phase 3: Enhance .env generation

Enhance `generateEnvContent()` in `deploy/route.ts` and the scaffold in
`harness.ts` to include:

- Commented sections for all supported platforms (not just the one selected)
- `PYTHONUNBUFFERED=1` (already in Dockerfile but good to document)
- Security notes about `GATEWAY_ALLOW_ALL_USERS` and allowed-users lists
- Debug flags (`WEB_TOOLS_DEBUG=false` etc.)

### Phase 4: Wire it up

Both creation paths (`/api/harnesses/create` and `/api/setup/deploy`)
should use the shared template instead of their inline generators. This
eliminates the current duplication where two files generate slightly
different configs.

### Phase 5: Expose in UI (future)

The setup wizard could show an "Advanced Config" expandable section letting
users toggle opinionated defaults before deployment. Not blocking for the
initial implementation.

## The Opinionated Default Config (Full)

```yaml
# Hermes Agent — generated by Swarm Map
# Full reference: https://github.com/NousResearch/hermes-agent/blob/main/cli-config.yaml.example

model:
  provider: ${provider}
  default: ${primaryModel}
  # fallback: ${fallbackModel}  # uncomment if set

# --- Context compression (prevents OOM on long conversations) ---
compression:
  enabled: true
  threshold: 0.50
  target_ratio: 0.20
  protect_last_n: 20
  summary_model: "google/gemini-3-flash-preview"

# --- Persistent memory (carries knowledge across sessions) ---
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200
  user_char_limit: 1375
  nudge_interval: 10
  flush_min_turns: 6

# --- Session reset (messaging platforms — prevents runaway token costs) ---
session_reset:
  mode: both
  idle_minutes: 1440
  at_hour: 4

group_sessions_per_user: true

# --- Agent behavior ---
agent:
  max_turns: 60
  verbose: false
  reasoning_effort: "medium"

# --- Voice transcription (pre-installed in Docker) ---
stt:
  enabled: true
  local:
    model: "base"

# --- Terminal ---
terminal:
  backend: "local"
  cwd: "."
  timeout: 180

# --- Skills ---
skills:
  creation_nudge_interval: 15

# --- Display ---
display:
  compact: false
  tool_progress: all
  interim_assistant_messages: true
  streaming: true

# --- Platform toolsets (explicit defaults) ---
platform_toolsets:
  cli: [hermes-cli]
  telegram: [hermes-telegram]
  discord: [hermes-discord]
  signal: [hermes-signal]
  slack: [hermes-slack]
```

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Compression model requires OpenRouter/Nous API key | Falls back gracefully; document that compression needs a second key if primary provider isn't OpenRouter |
| `stt.local` model download on first voice message | "base" is ~150MB, downloads once; acceptable for Docker where disk is cheap |
| Opinionated defaults conflict with upstream changes | Template references upstream example version; `/update` in the template header |
| Users confused by large config file | Sections are clearly commented; minimal users can delete sections they don't need |

## Files to Change

- `lib/services/harness.ts` — fix volume mount (`/opt/data`), add `command: gateway`, use shared template
- `app/api/setup/deploy/route.ts` — use shared template for `generateConfigYaml()`
- `lib/templates/config.yaml.ts` — new file, the opinionated template
- Tests for the template generation

## Dependencies

- None — this is internal to hermes-swarm-map
- Hermes upstream config.yaml format is stable (backwards-compatible, additive)
